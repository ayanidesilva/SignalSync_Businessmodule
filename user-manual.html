1. Introduction

SignalSync is a Business-to-Government (B2G) smart traffic management platform designed to address Colombo's chronic urban congestion. The system replaces outdated fixed-time traffic signals with AI-driven adaptive control, using the city's existing CCTV infrastructure rather than requiring expensive new hardware installation.

The platform serves two distinct audiences: traffic management officers who use a secure operations dashboard to monitor junctions, manage incidents, and activate emergency corridors; and the general public, who access a citizen-facing portal for live congestion maps, incident alerts, and route guidance.

How to access the prototype: Run npm run dev in the signalsync/ directory. The site serves at http://localhost:5173. No login is required — navigate directly to any page via the URLs shown in each section below.

2. Site Structure Overview

The prototype is divided into three access zones:

  • Public Portal — five pages accessible to any citizen, no authentication required.
  • Operations Platform — seven pages for trained traffic management officers (simulated; no real login gate in the prototype).
  • Administration — two pages for system administrators managing users and AI configuration rules.

The public-facing pages use a lighter navy/white layout with a sticky navigation bar. Platform and admin pages use a dark dashboard layout with a collapsible sidebar, breadcrumb trail, and a persistent live-incident alert banner.

6. Prototype Limitations & Planned Future Implementation

Note: The limitations listed below reflect deliberate design scope decisions and time/technical constraints of this coursework task — not failures of execution. Each represents a planned feature with a clear implementation path.
Planned

Authentication & Session Management

The prototype allows direct navigation to all platform and admin pages without login. Full implementation requires JWT-based authentication, session expiry, and route guards in React Router redirecting unauthenticated users to a login page.

Planned

Live Data & WebSocket Feeds

All metrics, junction states, and CCTV data are static mock arrays. Production requires a real-time WebSocket connection to junction controllers, updating the UI every 30 seconds. KPI metrics would be served from a time-series database.

Planned

Real CCTV Video Streams

The CCTV page simulates feeds with styled placeholder tiles. Full implementation would embed authenticated RTSP-to-HLS video streams and overlay real-time AI detection bounding boxes from the edge processing unit.

Planned

Interactive GIS Map

The Congestion Map uses an SVG background image with fixed-position markers. Production would integrate Mapbox GL JS or Leaflet with real GPS coordinates, live polyline corridor overlays, and tile-based street maps.

Planned

Report Export

The Export button on the Reports page shows a format dropdown but does not generate files. Implementation requires server-side PDF generation (e.g. Puppeteer or WeasyPrint) and CSV serialisation of filtered query results.

Planned

Incident & Rule Write Operations

Forms (Create Incident, Create User, Create Rule Profile) submit data but do not persist it — the page state resets on reload. Full implementation requires a REST API backed by a relational database (e.g. PostgreSQL) with change events broadcast to connected clients.

Planned

Email / SMS Notification Dispatch

When an incident is created, a success message states "Notifications dispatched." In production, this would trigger Twilio SMS alerts to registered field officers and email advisories to subscribed citizens via a notification microservice.

Planned

AI Model Confidence Feedback Loop

The CCTV page shows static detection confidence percentages. The production system would allow operators to flag incorrect detections, feeding corrections back into a model retraining pipeline to improve accuracy over time.
